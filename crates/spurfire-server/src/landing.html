Inside a lobby

A temporary posse, not a game server.

The control plane introduces the riders and coordinates lifecycle. Once connected, movement, shots, probes, and snapshots travel between players instead of bouncing through Ottawa.

1

Create

A lobby record fixes capacity and expiry, then prepares an isolated network boundary.

2

Join once

Each rider gets one short-lived, non-reusable enrollment credential. Organization credentials stay server-side.

3

Connect

RustScale gives every peer a private endpoint and chooses direct UDP, peer relay, or DERP.

4

Elect

Fresh route, latency, loss, and device measurements produce the same authority winner on every peer.

5

Ride & retire

Authority migrates after host loss. Match end deletes the dedicated lobby network and ephemeral nodes.

⚑
Public preview: zero Tailscale mutations

Ottawa currently demonstrates the lobby API in dry-run mode. Public Create/Join UI and real provisioning stay disabled until identity, abuse controls, and encrypted cleanup recovery are deployed.

Lobby metadata

What the lobby knows—and what it does not.

The prototype uses pseudonymous, client-generated IDs. It does not yet operate an account system, and that is also why the public mutation API is not enabled.

Kept for lobby operation

Small, inspectable records needed for roster, election, expiry, and cleanup.

  • Lobby and player UUIDs plus display names
  • Horse choice, wire version, and join state
  • Route counts, RTT, jitter, loss, and timestamps
  • Authority result, lifecycle state, and non-secret cleanup receipts

Never placed in lobby state

Secrets and unrelated identity data stay outside gameplay records.

  • No email, password, social profile, or advertising ID
  • No organization OAuth or child-tailnet secret
  • No durable auth key, bearer token, or node private key
  • No permanent server copy of peer-to-peer gameplay packets
